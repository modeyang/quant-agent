# Quant Agent v2

> 面向个人交易者的 A 股量化交易操作系统  
> 核心公式：`Agent = LLM + Deterministic Modules + Data + Memory`

---

## 1. 设计目标

本项目不是高频量化引擎，也不追求与专业量化团队拼速度。目标是为个人交易者构建一个 **盘后主决策、盘中轻触发、人工确认执行、自动复盘归档** 的 Harness Engineering Agent。

核心价值：
- 提高候选筛选质量
- 强化交易纪律和风控
- 降低盘中噪音干扰
- 沉淀可复用的交易记忆

---

## 2. 系统边界

### 2.1 资产范围
- A 股现货
- 场内 ETF
- 可转债

### 2.2 交易风格
- 主风格：短波段，持有周期 `2-10` 个交易日
- 辅风格：中波段，持有周期 `2-8` 周
- 策略定位：事件驱动发现机会，趋势结构确认进出场

### 2.3 Phase 1 执行权限
- Agent 生成交易计划和订单草案
- 用户人工确认后执行
- 不允许自由自动下单

### 2.4 盘中监控范围
- 当前持仓
- 观察池
- 当日候选池

不做全市场高频扫描，不做 T0/打板系统。

### 2.5 资金池设计
- 主池：A 股现货 + ETF
- 辅池：可转债

主池和辅池独立管理、独立考核、独立风控。

---

## 3. MVP 产品定义

MVP 只做最小可用交易闭环：

1. 盘后生成次日交易计划
2. 盘前输出作战简报
3. 人工确认后一键执行
4. 盘中轻监控计划是否失效或强化
5. 收盘后自动复盘和记忆写入

MVP 不做：
- 全自动自由下单
- 高频/低延迟撮合
- 全市场 Tick 级扫描
- 复杂回测平台
- 社媒情绪主导的交易决策

---

## 4. Harness Engineering 原则

### 4.1 角色分工
- LLM：解释、总结、排序、生成计划、复盘归因
- 确定性代码：指标计算、打分、仓位、风控、审批、执行、对账
- 数据层：提供统一事实
- Memory：存储结构化经验和证据链

### 4.2 模块化原则
- 高内聚、低耦合
- 业务闭环优先于技术分层
- Skill 作为编排壳，不承载核心交易规则
- 核心交易规则必须放在可测试 Python 模块中

### 4.3 Skill 复用准入标准
一个现成 skill 只有满足以下条件才能进入主链：
- 数据源明确
- 运行环境明确
- 有真实代码而不只是 prompt
- 有结构化输出
- 不强依赖固定飞书/账号/绝对路径
- 与本项目交易风格匹配

不满足标准的 skill 只能作为代码 donor 或参考实现。

## 5. 日内运行 DAG

### 5.1 盘前与盘中轻触发链

| 时间 | 节点 | 作用 |
|------|------|------|
| `08:30` | 作战准备 | 隔夜新闻、公告、观察池初稿 |
| `08:45` | 系统自检 | 交易日历、账户连接、订单状态、数据源健康度 |
| `09:12` | 盘前简报 | 持仓风险、候选池、禁做清单 |
| `09:20` | 集合竞价快照 | 优先级修正，不直接追单 |
| `09:26` | 开盘计划冻结 | 标记可执行试仓/仅观察/只卖不买/禁做 |
| `09:45` | 第一确认点 | 决定是否执行首笔试仓 |
| `10:30` | 上午趋势复核 | 删除走弱计划，提升强化标的优先级 |
| `11:35` | 午盘前归档 | 记录上午异动和计划失效原因 |
| `13:05` | 午后复核 | 检查资金回流、板块切换、持仓风险变化 |
| `14:00` | 隔夜预决策 | 判断继续持有、放弃计划或等待尾盘 |
| `14:50` | 尾盘执行窗口 | 执行按计划的建仓、减仓、风控动作 |
| `14:57` | 收盘前快照 | 固化收盘结构，喂给盘后链路 |

### 5.2 盘后主决策链

| 时间 | 节点 | 作用 |
|------|------|------|
| `16:00` | 收盘入库 | 行情、板块、情绪、持仓表现入库 |
| `17:00` | 主题与龙头识别 | 识别板块轮动、龙头强度、候选扩缩 |
| `18:30` | 多源融合与打分 | 事件、技术、资金、持仓上下文综合评分 |
| `19:00` | 复盘与归档 | 交易复盘、失误归因、记忆写入 |
| `20:00` | 次日计划草案 | 输出买卖计划、试仓顺序、风险提示 |

设计原则：盘中不重算全市场，重计算放到盘后。

## 8. 数据架构

### 8.1 单一事实源
- 结构化事实主存储：`SQLite`
- 人类可读产物：`Markdown / JSON`
- 订单、成交、持仓、账户快照只信数据库，不信 LLM 文本

### 8.2 统一主键与字段
所有记录至少带：
- `trade_date`
- `ts`
- `symbol`
- `asset_type`
- `source`
- `run_id`

### 8.3 资产主数据

`asset_master` 统一管理：
- 股票 / ETF / 可转债代码
- 交易所
- lot size
- T+ 规则
- 涨跌停规则
- 交易时段
- 行业 / 概念
- 正股映射（转债）

- `watchlist`

## 9. Memory 与多源消息门控

### 9.1 Memory 分层
- `market_memory`：市场风格、主线、情绪
- `thesis_memory`：个股/ETF/转债关注逻辑
- `trade_memory`：每笔交易的计划、执行、结果
- `postmortem_memory`：大赚大亏复盘、错误模式
- `playbook_memory`：高胜率形态和禁用环境

### 9.2 多源消息融合流水线
1. `raw_ingest`
2. `normalize`
3. `cluster`
4. `score`
5. `gate`
6. `dispatch`

### 9.3 建议表结构
- `raw_message`
- `event_cluster`
- `evidence_item`
- `memory_entry`
- `conflict_queue`
- `entity_link`

### 9.4 记忆门控规则
每条消息按以下维度打分：
- `source_reliability`
- `source_independence`
- `freshness`
- `portfolio_relevance`
- `novelty`

写入规则：
- `score >= 0.85`：`confirmed memory`
- `0.65 <= score < 0.85`：`provisional memory`
- `< 0.65`：只进短期缓存

冲突时：
- 不覆盖旧记忆
- 进入 `conflict_queue`
- 等待更高等级来源确认

优先级：
`交易所/公司公告 > 官方终端/官方数据 > 主流财经媒体 > 聚合站 > 社媒`

## 10. 策略与仓位规则

### 10.1 选股与交易逻辑
- 事件驱动负责发现机会
- 趋势结构负责确认进出场
- 板块强度和资金确认作为过滤层

### 10.2 主池规则
- 常态持仓：`3-5` 只
- 单票最大目标仓位：`25%`
- 首笔试仓：目标仓位的 `1/3`
- 不对亏损仓补仓
- 加仓只允许发生在：
  - 已浮盈
  - 走势继续强化
  - 板块/主题逻辑未破坏

### 10.3 辅池规则
- 可转债独立管理
- 优先考虑流动性、溢价率、正股联动
- 不与主池共享同一套仓位权重

### 10.4 卖出纪律
- 第一层：硬止损保命
- 第二层：趋势结构破坏退出
- 卖出信号包括：
  - 固定亏损阈值触发
  - 关键位失守
  - 板块退潮
  - 量价结构恶化
  - 计划失效

---

## 11. 风控与审批矩阵

### 11.1 风控层级
- 账户级：总仓位、单日亏损、最大回撤
- 资金池级：主池/辅池仓位上限
- 标的级：单票仓位、流动性、涨跌停规则
- 订单级：最小成交量、价格偏离、重复下单保护

### 11.2 默认风控原则
- 初始阶段总仓位上限可配置，默认不满仓
- 新开仓必须人工确认
- 非白名单标的不允许自动进入订单提交阶段
- 异常状态触发 `kill switch`

### 11.3 审批矩阵

| 动作 | 默认策略 |
|------|---------|
| 新开仓试仓 | 必须人工确认 |
| 加仓 | 必须人工确认 |
| 主动减仓 | 可以半自动生成建议，执行前确认 |
| 风险止损卖出 | 默认建议人工确认；后续可升级为白名单自动 |
| 撤单 | 人工确认 |
| 调整参数/风控阈值 | 人工确认 |

## 12. 执行状态机

执行链必须具备明确状态和幂等性。

### 12.1 状态流
`planned -> draft_order -> approved -> submitted -> partial_fill -> filled / canceled / rejected -> reconciled -> reviewed`

### 12.2 必须实现的控制点
- 同一 `plan_id` 不得重复提交
- 订单提交失败可重试，但必须带幂等键
- 成交与持仓必须对账
- 盘后复盘必须基于对账后的真实成交

### 12.3 对账对象
- 委托状态
- 成交明细
- 持仓快照
- 账户资产快照

## 14. MVP 验收标准

满足以下条件即视为 MVP 可用：
- 收盘后可自动生成次日交易计划
- 盘前可生成作战简报
- 用户可人工确认后一键执行
- 执行结果可自动回写订单、成交、持仓、账户快照
- 收盘后可自动生成复盘
- 盘中仅监控持仓、观察池、候选池
- 全链路不依赖飞书作为唯一输出渠道

## 17. 风险声明

- 本系统首先是交易操作系统，其次才是“智能分析器”
- LLM 不能直接成为最终风控器和下单器
- 所有实盘动作必须具备审计线索和人工接管能力
- 策略结论不构成投资建议，实盘需自行承担风险
